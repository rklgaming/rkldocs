Gear · Forge · Crafting · Magic

RKL Suite — Developer Docs

Connected FiveM resources for RPG servers: wearable gear with stats and durability, a forge for upgrades, enchants and sockets, an XP-based crafting system, and a full spell & skill layer. Everything runs on ESX Legacy, QBCore and Qbox through auto-detecting bridges.

rklequipment — gear, stats, combat rklforge — upgrade · enchant · socket rklcrafting — recipes & XP spell system — 5 magic resources effects & FX — status & particles rklinterface — HUD & class build

Overview

Each resource is independent but they share data. The forge reads and writes the equipment metadata, so equipment is the core the others build on.

ResourceDoesOwns
rklequipmentEquip gear, apply stats, durability, combat combosplayer_equipment
rklforgeUpgrade / enchant / socket equipped gearedits equipment metadata
craftingCraft items, levels & XPslv_crafting_data
Load order matters

The forge edits equipment metadata directly, so it must start after equipment. Full order is in the next section.

Install & server.cfg

Required on every framework

ox_lib, ox_inventory, and oxmysql. No es_extended is required for QBCore/Qbox — the bridge detects the framework automatically (qbx_core → Qbox, qb-core → QBCore, es_extended → ESX Legacy).

Ensure order

server.cfg
ensure oxmysql
ensure ox_lib
ensure ox_inventory

# your framework (one of)
ensure es_extended      # or qb-core / qbx_core

ensure rklequipment     # core — start first
ensure rklforge         # AFTER equipment
ensure rklcrafting
ensure rklskilltree     # class XP / skill nodes
ensure rklbinder        # card binder (uses forge_card)
ensure rklrepair        # durability repair NPC

# effects (needed for combat visuals & damage numbers)
ensure rklparticleseffect
ensure rklstatuseffect

# spell system
ensure rklgrimoire
ensure rklskillsystem
ensure rklspellmanage
ensure rklspellbook
ensure rklskillcheck

# HUD (reads skilltree + forge)
ensure rklinterface

Currency — pick what players pay with

Costs that the scripts call "awaken coin" are not hard-wired to an item. Set Config.RKLCurrency in the resource's config.lua (an open file, editable under escrow) and the bridge routes every charge and balance check for you — same behaviour on ESX Legacy, QBCore and Qbox.

config.lua
Config.RKLCurrency = { type = 'item', item = 'awaken_coin' }  -- ox_inventory item
-- Config.RKLCurrency = { type = 'money' }                     -- cash
-- Config.RKLCurrency = { type = 'bank'  }                     -- bank balance
-- Config.RKLCurrency = { type = 'account', account = 'black_money' }
typeESX LegacyQBCore / Qbox
itemox_inventory itemox_inventory item
moneygetMoney()money.cash
bankgetAccount('bank')money.bank
accountnamed ESX accountnamed money type
Leaving it out is safe

If Config.RKLCurrency isn't defined, nothing changes: an ESX server's own getAwakenCoin() is used when it exists, otherwise the awaken_coin ox_inventory item. Defining it always wins, so your choice is honoured on every framework.

Per-shop currency in rklequipment

Equipment shops already choose their own currency with currencyType in Config.Shops — set it to money, bank, any ox_inventory item name, or awaken_coin to follow the Config.RKLCurrency setting above.

Resource names

Exports and cross-resource events use fixed names, and a FiveM export only resolves if the folder name matches the caller. The whole suite is branded rkl* — keep these folder names exactly:

Folder nameUsed by
rklequipmentexports['rklequipment'] · rklequipment:* events
rklforgeexports['rklforge'] · rklforge:* events
rklcraftingrklcrafting:* events & callbacks
All references are rkl

Every export, event, callback and cross-resource sync uses the rkl names above, matching the folders. NUI uses GetParentResourceName(), so the UI follows the folder name automatically. The database tables (player_equipment, slv_crafting_data) keep their original names so existing data is untouched.

Optional integrations

Three resources outside this suite are used when present: rklzones (safe zones), rklcompanion (pet detection) and rklspawnmonster (monster lookup). rklskilltree and rklforge are optional to each other in the same way.

Every one of these calls is wrapped, so a missing resource can never throw. You can also switch them off explicitly:

config.lua
Config.Integrations = {
    ['rklzones']        = true,  -- fallback: never in a safe zone
    ['rklcompanion']    = true,  -- fallback: not a companion
    ['rklspawnmonster'] = true,  -- fallback: no monster
    ['rklskilltree']    = true,  -- fallback: level 0
}
ValueBehaviour
trueUse it, but only when the resource is actually started
falseNever call it — the fallback value is used
omittedSame as true, so older configs keep working

Shared state bag

LocalPlayer.state.playerClass (and playerSubclass) is how the player's class travels between resources. rklskilltree sets it; rklinterface and rklequipment read it. It is framework-independent, so it behaves the same on ESX, QBCore and Qbox.

Start order matters

rklforge writes to equipment metadata and rklinterface reads forge stats, so start rklequipment first, then rklforge, then the rest. The full order is in Install & server.cfg.

Character class columns

The skill tree stores the class, sub-classes and rank on the character row. Run the block that matches your framework — the SQL file has both, with the QBCore/Qbox one commented out.

ESX Legacy
ALTER TABLE `users`
    ADD COLUMN IF NOT EXISTS `class`       VARCHAR(64) DEFAULT NULL,
    ADD COLUMN IF NOT EXISTS `sub_classes` TEXT        DEFAULT NULL,
    ADD COLUMN IF NOT EXISTS `rank`        VARCHAR(8)  DEFAULT 'E';
QBCore / Qbox
ALTER TABLE `players`
    ADD COLUMN IF NOT EXISTS `class`       VARCHAR(64) DEFAULT NULL,
    ADD COLUMN IF NOT EXISTS `sub_classes` TEXT        DEFAULT NULL,
    ADD COLUMN IF NOT EXISTS `rank`        VARCHAR(8)  DEFAULT 'E';
The identifier column

Scripts store whatever the bridge returns — the ESX identifier on ESX Legacy, the citizenid on QBCore/Qbox. Both fit VARCHAR(64), so no schema change is needed per framework. Moving frameworks on the same database means old rows won't match, which is normal for any migration.

Open files — combos & language

Two files ship outside the escrow so server owners can edit them with the resource still encrypted. Both are listed in escrow_ignore.

language.lua

Every player-facing string lives in a Lang table. Translate values freely; keep any %s placeholder — it's filled at runtime (for example the item name in a "broke" message).

language.lua
['armor_broken_desc'] = 'Your %s broke! Its bonus stats are gone.',

The helper is called _L('key') (not _U) so it never clashes with ESX's built-in translator.

combos.lua

Each weapon type's attack animation sequence is data only — change the dictionary, clip, timing or number of stages without touching logic.

combos.lua
Config.Combos.Katana = {
    [0] = {dict = 'kp_wm_gs_slash_01', anim = 'kp_wm_gs_slash_01_clip', wait = 700, next = 1},
    [1] = {dict = 'kp_wm_gs_slash_02', anim = 'kp_wm_gs_slash_02_clip', wait = 700, next = 2},
    -- last stage loops back with next = 0
}

Built-in types: BigSword, Staff, Unarmed, DualDagger, Katana, DualKatana.

Add a brand-new weapon type

The combo dispatch is data-driven: any weapon type that isn't built in is played by the generic runner — no escrowed edits needed.

  1. Give the weapon item w_type = "yourtype" in config.lua.
  2. Define Config.Combos.yourtype in combos.lua.
combos.lua — custom type
Config.Combos.spear = {
    [0] = {dict = 'melee@large_wpn@streamed_core', anim = 'plyr_takedown_front_attacker',
           wait = 900, flag = 1, next = 1, sound = 'hitflesh', damageRange = 2.5},
    [1] = {dict = 'melee@large_wpn@streamed_core', anim = 'plyr_takedown_rear_attacker',
           wait = 1100, flag = 1, next = 0, freeze = true,
           vfx = {dict = 'scr_sum2_hal', part = 'scr_sum2_hal_rider_weak_green',
                  attachType = 'weapon_tip', loop = true,
                  color = {r = 0.2, g = 0.6, b = 1.0}, scale = 0.5, duration = 700,
                  off = {x = 0.0, y = 1.0, z = 0.0}}},
}

Optional per-stage fields

flag, freeze, blendIn, blendOut, wait, next, sound, damageRange, and vfx (a particle table — add weapon-tip effects straight from the open file).

Optional per-type tuning

combos.lua
Config.ComboSettings = {
    spear = { speedFactor = 0.03, speedCap = 0.30, minWait = 300, comboTimer = 5, slot = 'right_hand' },
}
Built-ins are untouched

The six built-in combos keep their own hand-tuned behavior (Staff casts priest/mage effects, DualDagger freezes, BigSword has VFX + attack speed). The generic runner is used for custom types only.

Effects & FX

Two shared layers used across combat and magic: a status-effect system (poison, burn, slow, blind, stun, freeze) with screen and world VFX, and a particle / damage-number layer that other resources fire to show hits and effects.

Shared by the suite

The FX layer handles rkldamageped:victim — the same event the equipment combos fire to damage NPCs and show floating numbers. Keep it running or combat visuals won't appear.

Class stats live in Config.BaseStatus (base health, HP per level, and per-class modifiers for mage, priest, ranger, and more) and Config.Abilities. Database tables: slv_player_job_points, slv_player_skills, slv_skills, slv_skill_trees.

Hunger & thirst are framework-aware

On ESX the bars are driven by esx_status. On QBCore and Qbox those events never fire, so the HUD reads the same values from player metadata instead — via hud:client:UpdateNeeds, on player load, and with a slow poll as a catch-all. Class comes from a state bag

The HUD reads LocalPlayer.state.playerClass (and playerSubclass). Whatever sets the class — a class-select resource, for example — should set that state bag so the HUD picks it up.

rklrepair durability NPC

Repair NPCs that restore durability on equipment items. Each NPC accepts certain categories and only lists gear below its damage threshold. No database, no exports — it works directly on ox_inventory metadata.

How it meets rklequipment

Both resources use the same ox_inventory item, equipment, and the same durability fields. Repairing writes metadata.stats.durability (the equipment system's source of truth) and mirrors it to metadata.durability so the ox_inventory bar agrees.

Unequip before repairing

Equipping moves an item out of ox_inventory and into the player_equipment table, so equipped gear isn't in the bag at all. The repair list therefore only shows unequipped items: unequip → repair → equip again. The restored durability carries over on re-equip, so no refresh event is needed.

Both are server-validated — the item is re-read server-side and the cost recomputed from config, so a tampered UI can't repair for free.

Loot luck from equipped gear

Boost a drop chance using the player's total luck (server side):

your loot script — server
local function rollDrop(src, baseChance)
    local luck = exports['rklforge']:GetPlayerLuck(src) or 0
    local chance = math.min(100, baseChance + luck * 0.5)
    return math.random(100) <= chance
end

Troubleshooting

An export returns nil

The folder name must match the export call — see Resource names. exports['rklequipment'] only works if the folder is rklequipment.

Forge doesn't apply stats to gear

Check the ensure order — the forge must start after equipment, and both need oxmysql, ox_lib, ox_inventory running first.

Currency isn't deducted on QBCore / Qbox

Create an awaken_coin ox_inventory item, or change the currency in config to money / bank / another item.

A combo plays no animation

The animation dictionary/clip must exist (a streamed .ycd or a base-game dict). A missing dict means the clip won't play even though the combo logic runs.
